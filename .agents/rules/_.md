### 1. Comportement Général et Stabilité
- **Langue** : Toutes tes réponses dans le chat doivent être rédigées en **français**. En revanche, **TOUS** les commentaires que tu ajoutes dans le code source doivent être rédigés en **anglais**.
- **Ton et Communication** : Garde toujours un ton strictement professionnel et formel. Évite toute familiarité, sarcasme, humour inapproprié ou utilisation excessive d'émojis (pas de smileys tels que 😂).
- **Stabilité (Vérification)** : Après chaque modification ou bloc de modifications dans le code, tu DOIS obligatoirement lancer une vérification pour t'assurer que le projet compile sans erreur (e.g. via la commande 'zsh -lc "./bin/build"').

### 2. Architecture et Dépendances
- **Règle Stricte des Dépendances** : L'arborescence du projet obéit à une architecture en couches étanche. Le dossier `Core` ne doit JAMAIS importer quoi que ce soit de `Infra` ou de `Inter` (exception : on autorise `Core` à importer du code `Infra` ou `Inter` SI l'import provient d'un fichier lui-même situé dans un sous-dossier `Infra/` ou `Inter/` au sein de `Core`). Le dossier `Infra` ne doit JAMAIS importer quoi que ce soit de `Inter`.
- **Résolution des Modules (Dossier `_`)** : Le répertoire `_` sert uniquement au groupement visuel des fichiers et DOIT être ignoré dans la déclaration des noms de modules Purescript. *(Exemple : le fichier situé physiquement dans `Core/Feat/UserAccount/_/Lifecycle.purs` recevra la déclaration de module `Core.Feat.UserAccount.Lifecycle`)*.
- **Variables d'environnement / Config** : L'utilisation de `lookupEnv` (de `Node.Process`) est formellement proscrite. Pour récupérer des valeurs de configuration ou d'environnement, utilise systématiquement la configuration centralisée (`InternalConfig` ou `PublicConfig`).

### 3. Syntaxe, Alias et Prelude Personnalisé
- **Nommage des Variables** : Les noms de variables ne doivent jamais être en snake_case. Utilise toujours le camelCase (ex: `myVariable` et non `my_variable`).
- **Interdiction du Prelude Standard** : N'importe JAMAIS le module `Prelude` standard de PureScript, sauf en cas d'absolue nécessité technique.
- **Utilisation de Proem** : Importe TOUJOURS le prelude personnalisé du projet via `import Proem`. Si tu rencontres des conflits de nommage avec les éléments HTML de Halogen, utilise la syntaxe d'exclusion : `import Proem hiding (div, top)`.
- **Alias Mathématiques Monadiques** : Utilise autant que possible les opérateurs Unicode personnalisés exportés par `Proem` (ou définis dans `Utils`) pour alléger le code :
  - `ʌ` pour `liftEffect`
  - `ø` pour `void`
  - `η` pour `pure`
  - `κ` pour `const`
  - `??⇒` au lieu de `fromMaybe` (ex: `maybeValue ??⇒ defaultValue`)
  - `∀` au lieu de `forall`
  - `?` et `↔` au lieu de `if ... then ... else ...` (ex: `condition ? ifTrue ↔ ifFalse`)
  - `ι` au lieu de `unit`
  - De manière générale, tu DOIS privilégier les raccourcis et symboles fournis par le projet dès que l'occasion se présente. L'utilisation systématique de ces alias est obligatoire pour alléger la lecture.
- **Pattern Matching de Records Prioritaire** : Au lieu d'utiliser l'accès par point (`s.abc`), privilégie systématiquement la déstructuration/pattern matching direct sur les records dans les arguments de fonction ou les `case` (ex: `{ abc }`). Si tu as également besoin d'une référence au record entier, utilise l'alias complet `s@{ abc }`.
- **Centralisation des Types Anonymes (Alias)** : Évite les répétitions de records de type anonymes complets (ex: `{ src :: String, format :: Format }`) dans diverses signatures, sauf en cas de besoin de row polymorphism. Préfère toujours les extraire dans un alias de type dédié (ex: `type Thumb = { src :: String, format :: Format }`) pour rendre la lecture plus sémantique et éviter la duplication.

### 4. Lisibilité et Découpage (Code as Documentation)
- **Factorisation des logiques touffues** : Face à une fonction longue et complexe impliquant diverses imbrications (`case`, `let`, logique conditionnelle mixée avec de la vue), tu DOIS systématiquement isoler et extraire ces détails d'implémentation dans des sous-fonctions (typiquement dans la clause `where` ou au niveau du module).
- **Typage explicite des sous-fonctions** : Toute fonction définie dans un bloc `where` DOIT systématiquement posséder sa propre signature de type. Cela permet d'améliorer la lisibilité globale et d'utiliser ce typage explicite comme une "documentation-code" sur le rôle de la sous-fonction.
- **Style déclaratif** : L'objectif est que le corps de la fonction principale devienne déclaratif et se lise intuitivement comme une documentation de haut niveau (ex: `item id (thumbFormat article) (extractTitle article)`). Le "comment" doit être déporté et masqué derrière des noms de fonctions explicites pour réduire la charge cognitive et éviter la duplication.
- **Pas de "Magic Numbers"** : Extrais systématiquement les nombres magiques ou les constantes arbitraires (limites, tailles, etc.) dans des variables globales clairement nommées et typées au niveau du module (ex: `maxSearchZoneLength :: Int`). Cela documente leur intention et facilite leur ajustement futur.

### 5. Frontend et CSS
- **Compatibilité Navigateur** : Toute solution CSS, comportement UI ou API Web proposée DOIT bénéficier d'un large support sur l'ensemble des navigateurs (y compris les versions un peu plus anciennes de Firefox, Safari, Edge, etc.). Ne propose JAMAIS de propriétés CSS expérimentales ou très récentes qui ne fonctionnent que sur des navigateurs ultra-modernes, à moins de garantir un fallback parfait.
